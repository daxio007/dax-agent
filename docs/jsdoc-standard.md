# JSDoc 规范

最后更新：2026-06-22

这份规范定义 DAX Agent 的方法级代码文档要求。它适用于 `src/**/*.ts`，目标是让阅读者不需要先追踪完整调用链，也能理解一个方法如何使用、承担什么职责，以及不能替调用方做什么。

## 强制覆盖范围

以下命名代码单元必须紧邻一段 JSDoc：

- 函数声明。
- 类和对象中的方法。
- 构造方法、getter 和 setter。
- 赋值给变量的箭头函数或函数表达式。
- 导出方法、公共方法、私有方法和关键内部 helper。

匿名内联回调不强制单独添加 JSDoc。它们应保持短小，并由所属的命名方法说明整体用途；复杂回调应提取为命名方法后再添加 JSDoc。

## 必填内容

每段方法 JSDoc 至少包含以下两个明确小节：

- `使用方法`：说明调用时机、主要参数或前置条件，以及调用方如何使用返回值。
- `作用`：说明方法在系统中的职责、产生的状态变化或业务意义。
- `@param`：为方法签名中的每一个参数逐项说明来源、含义、用途，以及重要的默认值或约束。

涉及路径、安全、外部请求、持久化、模型调用、执行或副作用的方法，还应包含：

- `边界`：说明方法不会做什么、失败方式、权限限制、数据范围或安全约束。

推荐格式：

```ts
/**
 * 使用方法：传入已经标准化的用户消息；调用方应使用返回值继续后续路由。
 *
 * 作用：把用户输入转换为内部决策，并记录本轮审计信息。
 *
 * 边界：这里只生成决策，不直接执行写入或 shell 命令。
 *
 * @param input 已经标准化的 Agent 输入，包含用户目标、约束和当前上下文。
 * @param options 可选的推理配置和依赖注入，主要用于测试或替换模型调用。
 */
function decide(input: AgentInput, options: DecideOptions = {}): AgentDecision {
  // ...
}
```

## 内容质量

JSDoc 应解释调用契约和设计意图，而不是把函数名翻译成一句中文。避免使用“处理数据”“返回结果”这类无法帮助阅读者判断行为的描述。

`@param` 不能只重复 TypeScript 类型。例如 `@param input AgentInput` 不合格；说明必须让阅读者知道该参数在当前方法中承担什么角色。

文档必须与当前实现一致。修改方法的输入、输出、副作用或边界时，应在同一次改动中更新 JSDoc。

## 自动检查

运行：

```bash
npm run check:jsdoc
```

检查脚本会扫描全部 `src/**/*.ts`，确认所有受规范约束的命名方法都存在 JSDoc，包含 `使用方法` 和 `作用`，并且每个形参都有带非空说明的 `@param`。检查失败时会输出文件、行号、方法名和缺失内容。

提交涉及 TypeScript 源码的改动前，至少应运行：

```bash
npm run check:jsdoc
npm run typecheck
npm run build
```
